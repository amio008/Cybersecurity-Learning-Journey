# Wireshark: Capturing and Analyzing Live HTTP Traffic

## Goal
Capture real HTTP traffic generated by my own browser and walk through every
layer of the OSI model as represented in the actual packet — not a simulated
network, but real traffic leaving and entering my machine.

## What I did
- Installed Wireshark and started a capture on my active network interface.
- Visited a plain HTTP (non-HTTPS) site so the traffic would be readable in
  cleartext.
- Applied the `http` display filter to cut out the noise (DNS, ARP, background
  traffic) and isolate the relevant request/response pair.
- Selected a request and response frame and expanded every layer in the
  packet details pane: Frame → Ethernet II → IPv4 → TCP → HTTP.
- Documented what each header field actually means and why it matters from a
  security perspective (not just "what protocol is this").

## Why this matters for security
This is the single most transferable skill in this whole exercise. Being able
to open a capture and immediately answer:
- What layer is this field even in?
- What port/protocol combination is in use?
- Is there anything here that shouldn't be visible in cleartext?
- What do response headers reveal about the backend (server software,
  security headers present or missing)?

...is exactly what you do during traffic analysis, incident response, and
basic recon/detection work. See `notes.md` for the full breakdown.

## Key takeaway
Plain HTTP exposes everything — headers, cookies, request bodies — to anyone
who can see the traffic. This capture made that concrete instead of
theoretical. It also reinforced why `X-Frame-Options`, `X-XSS-Protection`,
and similar response headers exist: they're a server telling the browser to
enforce certain security behaviors, and their absence is itself a signal
worth noting during a security review.
